1. Executive Summary

This document outlines the requirements for the Enterprise Role Form Configuration Suite, a comprehensive backend solution for our SaaS customers. Moving beyond basic settings, this suite will provide administrators with a guided, wizard-like experience to deeply customize every aspect of the "Add/Edit Role Form." Key capabilities include global feature toggles for AI and data management, company branding for exports, granular permission controls, full localization for a global workforce, and robust versioning with a complete audit trail.

This evolution is critical to capturing the enterprise market segment, offering unparalleled flexibility and control that allows large organizations to seamlessly integrate our platform into their existing global HR ecosystems.

Success will be measured by:

  • 75% adoption of the configuration suite by enterprise customers within one year.
  • Measurable reduction in customer support requests for form customizations by 95%.
  • Successful deployment in at least 5 multinational customers requiring localization and branding features.

2. Problem Statement & User Need

Enterprise customers operate at a scale and complexity that a one-size-fits-all form cannot accommodate. They have diverse global workforces, strict branding guidelines, complex security models, and established terminology. The inability for an administrator to self-serve these customizations creates friction, slows down adoption, and makes our product feel less like a tailored enterprise solution and more like a rigid tool.

User Persona: Enterprise System Administrator, Global HRIS Administrator.

User Need: Administrators require a powerful, centralized, and intuitive suite to configure the Role Form. They need to control features, apply corporate branding, define who can do what, adapt the form for different languages and cultures (including RTL), and manage these complex configurations over time with versioning and auditing to meet enterprise-grade compliance and governance standards.

3.9 Step 9: Review & Activate

A final summary step that provides a clear "diff" view before deploying changes.

  • Functionality: Displays a read-only, side-by-side comparison or highlighted summary of all changes made in the current session versus the currently active version. Additions will be shown in green, modifications in yellow, and deletions in red.
  • Final Confirmation: The admin can review this clear summary of their impact before clicking the final "Save and Activate" button, which will create a new version and make it live.

4. Implementation: Frontend & Backend Feature Mapping

Feature / CapabilityFrontend Responsibilities (Client-Side)Backend Responsibilities (Server-Side)
Overall Configuration Wizard
  • Build the main sequential wizard UI with "Next" and "Back" buttons.
  • Manage the local state of the wizard (which step is active).
  • Render the appropriate components for each step.
  • Create a secure endpoint accessible only to admins.
  • Provide an API to fetch the current, complete configuration state to populate the wizard on load.
Step 1: Global Settings
  • Render the on/off toggle switches for AI, Import/Export, Print Preview, and Tooltips.
  • Display the current state of each toggle based on data from the backend.
  • Send an API request when a toggle state is changed.
  • Store the boolean state (true/false) for each global feature toggle in the database per tenant.
  • Create API endpoints to `GET` the current state and `PUT` updates for these toggles.
Step 2: Branding
  • Create a file upload component for the company logo with client-side validation (file type, size).
  • Display a preview of the uploaded logo.
  • Render text input fields for header/footer content.
  • Send logo file and text to the backend upon saving.
  • Create an endpoint to handle the logo file upload and store it securely (e.g., in an S3 bucket).
  • Store the logo URL and header/footer text in the database per tenant.
  • Provide an API to retrieve the current branding assets.
Step 3: Form Structure & Field Config
  • Render a visual representation of the 9-step user form.
  • For each field, display controls to toggle visibility, edit the label, and edit the tooltip text.
  • For dropdowns, render a UI to add, edit, reorder, and deactivate options.
  • Render the UI for defining the skills proficiency scale.
  • Manage the state of these changes locally before they are saved.
  • Provide an API to `GET` all configurable fields and their current state (visible, label, tooltip, options).
  • Provide an API to `GET` and `PUT` the proficiency scale definition.
  • Create endpoints to `PUT` updates for field visibility, labels, and tooltip content.
  • Create `POST`, `PUT`, and `DELETE` (soft delete) endpoints for managing dropdown options.
Step 4: Custom Field Management
  • Build a UI for creating a new custom field (form with inputs for label, type, tooltip, etc.).
  • Render a list of all existing custom fields with options to edit or deactivate.
  • Perform client-side validation on the creation form.
  • Create a database table/collection to store custom field definitions per tenant.
  • Implement full CRUD API endpoints (`POST`, `GET`, `PUT`, `DELETE`) for managing custom fields.
  • Ensure the main Role Form API can incorporate and serve these custom fields.
Step 5: Localization & Cultures
  • Render a UI to add/remove languages for the tenant.
  • Display a two-column table for translating field labels, help text/tooltips, and dropdown options.
  • Implement the UI logic to dynamically switch the entire application to RTL when an RTL language is selected by the end-user.
  • Store all translations (key-value pairs) in the database, linked to a specific language and tenant.
  • Provide an API to `GET` all translatable strings and `PUT` updated translations.
  • Include a `direction: 'rtl'` flag in the user session/profile API when an RTL language is active.
Step 6: Permissions & Access
  • Build a UI for creating and managing roles (e.g., "HRBP").
  • Render a permission matrix (grid of checkboxes) for each role.
  • Send the updated permission set for a role to the backend when saved.
  • Create database tables to store custom roles and their associated permissions per tenant.
  • Implement CRUD APIs for managing these roles.
  • Enforce these permissions via middleware on all relevant data access endpoints for the end-user form.
Step 7: Versioning
  • Render a UI that lists all saved configuration versions.
  • Provide a button to "Activate" a specific version.
  • Display a modal asking the admin to name a new version upon saving changes.
  • Store entire configuration objects as versioned documents in the database.
  • Implement logic to flag one version as "active" per tenant.
  • Create an API to `GET` all versions and an endpoint to `POST` a new version or `PUT` to change the active version.
Step 8: Audit Log
  • Build a UI to display the audit log in a table format.
  • Implement filter controls (date range, user, action type).
  • Handle pagination for large logs.
  • Render an "Export to CSV" button.
  • Create a logging service that automatically records every change made through the configuration APIs.
  • Store immutable logs with all required details (timestamp, user, action, old/new values).
  • Provide a powerful query API to search and filter the logs.
  • Create an endpoint to generate and stream a CSV export of the filtered log data.
Step 9: Review & Activate
  • Create a read-only summary component that displays all pending changes from the current session.
  • Render the final "Save and Activate" button.
  • On click, trigger the API call to save the new configuration as a new version and make it active.
  • Create a single, transactional API endpoint that takes the entire pending configuration state, creates a new version, sets it to active, and logs the "Configuration Activated" event in the audit log.

5. Non-Functional Requirements

  • Security: The entire suite must be protected by a top-level admin role. The granular permissions defined in Step 6 must be strictly enforced on the frontend and backend.
  • Localization: The platform must support UTF-8 character sets. The front-end architecture must be built to support dynamic RTL/LTR switching based on the selected language.
  • Performance: All configuration pages must load in under 3 seconds. Activating a new configuration version should propagate to all users in under one minute.
  • Scalability: The system must support thousands of custom fields and dropdown options per enterprise tenant without performance degradation.
  • Data Integrity: Deactivating options/fields is preferred over hard deletes to ensure historical data remains intact. Rollbacks to previous configuration versions must not corrupt existing role data.
